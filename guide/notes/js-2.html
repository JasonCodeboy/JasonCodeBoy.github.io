# 作用域 Scopes，作用域链形成的原理，闭包形成的原理

var a1 = "1";
function test() {
  var a2 = "2";
  function fn() {
    console.log(a1);
    console.log(a2);
  }
  fn();
}

相信答案是一目了然的，a1 打印 '1'，a2 打印 '2'。那么为什么在函数 fn 内，可以访问到 a1 和 a2 呢，接下来就让我们一起来探索作用域的结构吧。

首先，大家看下图：

声明一个函数 fn，通过 console.dir(fn) 打印，其中有一个 [[Scopes]] 属性，这就是我们今天的主角作用域。

作用域中，存储着执行期上下文的集合，因为这个集合是呈链式结构，所以也称为作用域链。

那么，什么是执行期上下文呢？

当一个函数执行时，就会创建一个执行期上下文的内部对象 AO。当函数执行完毕后，它所产生的 AO 就会立即销毁。函数每次执行，所创建的 AO 都是独一无二的。

# 那么让我们以这种形式来解析下上面的代码：


var a1 = '1'
function test(){
    var a2 = '2'
    function fn(){
        console.log(a1)
        console.log(a2)
    }
    fn()
}
test()


全局预编译：
GO = {
    a1: '1',
    test: function test(){}
}

test函数内部预编译：
test.AO = {
    a2: '2',
    fn: function fn(){}
}

当全局预编译的时候，函数 test 被定义，根据控制台打印的结果，test 下会有一个 [[Scopes]] 的属性，其中存储着 0: Global。也就是：
test.[[Scopes]] = {
    0: Global
}

当函数 test 执行时，会创建一个独一无二的内部对象AO，也就是：
test.[[Scopes]] = {
    0: test.AO,
    1: Global
}

相信解析到这里，大家已经看出来了，这里的 test.AO 和 Global 就是上面预编译后的产物。

那么 test 函数内部，同样有一个 fn 函数定义：
fn.[[Scopes]] = {
    0: test.AO,
    1: Global
}

当 fn 函数执行时，会创建一个独一无二的AO，因为并没有声明，所以预编译后是个空对象。也就是：
fn.AO = { }
fn.[[Scopes]] = {
    0: fn.AO,
    1: test.AO,
    2: Global
}

然后在 fn 函数中，打印了 a1。那么 JS 引擎就会在当前作用域 fn.AO 中查找，有没有 a1 这个变量，结果显而易见，fn.AO是个空对象，并没有 a1。那么接下来，它就会沿着作用域链，依次查找 test.AO 和 Global，找到就立刻返回。
同样，打印 a2 也是如此。

那么总结来说，就是一句话：在哪个函数里面查找变量，就从哪个函数作用域链的最顶端，依次向下查找。

附上一张图方便理解： 了解了作用域和其查找变量的方式，那么对于闭包的产生是不是也有了一点头绪呢？

我们对上面的代码改变下，将 fn 函数 return 出去，并用 closure 接收。

function test() {
  var a = "1";
  function fn() {
    console.log(a);
  }
  return fn;
}
var closure = test();
console.dir(closure);

通过上图，我们可以明显的看到，作用域居然被保存到外部了。那么让我们一起来解析下，这其中到底发生了什么。

当 test 执行时：
test.[[Scopes]] = {
    0: test.AO,
    1: Global
}

test执行的同时，也是 fn 被定义，那么：
fn.[[Scopes]] = {
    0: test.AO,
    1: Global
}


当 test 函数执行完毕后，发生了2件事：
1，test.AO立即被销毁。
2，将 fn 的引用赋值给 closure


我们一步步来看：
test.AO 将被销毁的时候，test.[[Scopes]] 中对 test.AO 的引用也就断开了。
而同时，closure 保存了 fn 的引用，也就是说，当前的这个 test.AO 还有变量引用着，所以不会被垃圾回收。
从这里也可以看出，所谓的函数执行完毕后，执行期上下文的销毁，并不是真正的销毁，而是引用的断开。


继而我们 closure 执行，closure.[[Scopes]]中的最顶端，就会生成一个 fn.AO = {}，打印 a，就会触发，JS引擎从作用域链的最顶端依次往下查找。

这也就是闭包的整个形成过程了
